## 二、确定的技术方案

### 方案：高德API + 人工补充

```
┌─────────────────────────────────────────────────────────────────┐
│                         数据获取流程                             │
└─────────────────────────────────────────────────────────────────┘

Step 1: 调用高德API批量获取
        ↓
        搜索关键词：景点、古城、古镇、寺庙、公园...
        获取字段：id, name, location, address, type, rating, photos
        ↓
Step 2: 数据清洗
        ↓
        - 去除重复
        - 过滤非景点（如酒店、餐厅误入）
        - 筛选评分>4.0或热门景点
        ↓
Step 3: 人工补充业务字段
        ↓
        打开每个景点的携程/马蜂窝页面，补充：
        - area: 判断属于哪个区域
        - tags: 判断适合什么标签
        - visit_duration: 记录推荐游玩时长
        - brief: 摘抄一句话简介
        ↓
Step 4: 导出为JSON/导入数据库
```

### 工作量估算

| 步骤 | 时间 |
|-----|------|
| 高德API获取 | 10分钟（自动化脚本） |
| 数据清洗筛选 | 30分钟 |
| 人工补充50个景点 | 约3-4小时（每个5分钟） |
| **总计** | **约4-5小时** |

## 三、下一步工作

### 3.1 设计区域划分（area字段）

需要确定大理分为哪几个区域，原则：
- 同一区域内的景点可以一天内游玩
- 区域数量控制在5-8个

**初步规划**：
```
古城区      # 大理古城及周边
洱海东岸    # 双廊、挖色等
洱海西岸    # 才村、龙龛等
喜洲        # 喜洲古镇、海舌
苍山        # 苍山景区
剑川        # 沙溪古镇
其他        # 鸡足山等较远景点
```

### 3.2 设计标签体系（tags字段）

需要确定有哪些标签，原则：
- 标签数量控制在10个以内
- 每个标签下至少有5个景点

**初步规划**：
```
古镇        # 古城、古镇、历史街区
自然风光    # 山、湖、湿地、公园
民族文化    # 白族、寺庙、民俗
摄影        # 出片、网红打卡
美食        # 餐饮集中区
徒步        # 需要走路/爬山
小众        # 人少、冷门
亲子        # 适合带小孩
```

### 3.3 制作标注模板

创建一个Excel/JSON模板，包含：
- 高德API自动填充的字段
- 需要人工填写的字段
